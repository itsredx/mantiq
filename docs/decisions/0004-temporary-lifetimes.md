# 0004 - Temporary Lifetimes

## Context
Expressions often generate temporary, unnamed values (e.g., the intermediate string in `"a" + "b" + "c"`, or an anonymous list in `print([1, 2, 3])`). Without clear rules on when these temporary values are freed, the compiler risks memory leaks (if never freed), use-after-free bugs (if freed too early), and dangling references.

Even in `Nizam` (which employs a borrow checker) and `Mantiq` (which may rely on dynamic allocations or future RC/GC), a deterministic rule for the lifespan of an anonymous temporary must be established so that LLVM codegen knows exactly where to insert `drop` (`mantiq_free`) instructions.

## Decision
Mantiq and Nizam adopt **Statement-Level Temporary Lifetimes**.

### 1. Lifespan of a Temporary
A temporary value lives exactly until the end of the full statement (lexical expression) in which it was created.
- Once the statement completes execution (e.g., at the newline or semicolon), all temporaries created during its evaluation are dropped.
- Example: In `print("Hello " + name)`, the concatenated string temporary is dropped immediately after `print` returns, before the next line begins.

### 2. Return Values and Ownership Transfer (Moves)
If a temporary is the direct operand of a `return` statement (e.g., `return [1, 2, 3]`), its lifetime is **extended** and ownership is transferred to the caller. It is *not* dropped at the end of the return statement.
Similarly, if a temporary is assigned to a named variable (`let x = make_list()`), ownership is transferred to `x`. `x` will then be dropped at the end of its lexical block (via `genAutoDrops`).

### 3. References to Temporaries
- **Nizam (Strict Mode)**: Taking a reference/pointer to a temporary and storing it beyond the current statement is strictly forbidden. The borrow checker will flag this as a compilation error (`Borrowck Error: temporary value dropped while borrowed`). Temporaries can only be passed as arguments to functions if the function signature guarantees it does not store the reference.
- **Mantiq (Dynamic Mode)**: Mantiq handles dynamic types (`Any`) by boxing them. Currently, anonymous dynamic allocations must be tracked per-statement. Future iterations of Mantiq may employ Reference Counting (RC) for `Any` types, meaning the temporary will be decremented at the end of the statement, keeping the object alive only if a reference was stored internally.

### 4. Overwriting and Reassignment
When a variable is reassigned (`x = new_value`), the old value held by `x` is immediately dropped *before* the new value is stored, assuming the old value was owned and requires freeing.

## Implementation Implications
In `codegen.zig`, statement-level nodes (like `ExprStmt`, `PrintStmt`) must aggregate any anonymous heap allocations generated by their child `genExpr` calls. Before the statement finishes, the codegen must emit `mantiq_free` for these aggregated temporaries, mirroring the existing `genAutoDrops` behavior used for named block-scoped variables.
