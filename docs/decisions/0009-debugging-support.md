# 0009 - Debugging Support

## Context
Debugging generated low-level code (LLVM IR) or diagnosing runtime crashes (like out-of-bounds array access or division by zero) is highly challenging without a formal mapping from the intermediate representation back to the AST source location.

## Decision
Mantiq implements compiler and runtime instrumentation to preserve source coordinate context throughout code generation and execution.

### 1. Source Spans in LLVM IR
- **Comment-Based Mapping**: For every expression and statement generated by the compiler, `codegen.zig` writes an inline comment prefix containing the 1-based source line and column numbers, alongside the AST node type (e.g. `; Span: [row: 12, col: 5] - Node: VarDecl`).
- **Benefits**: This allows developers inspecting generated `.ll` files to immediately locate which line of the high-level `.mq` source file produced a given block of instructions.

### 2. Runtime Panic Source Location Tracing
- **Panic Instrumentation**: Rather than aborting blindly, runtime panic blocks (bounds checks and division checks) call a location-aware panic utility:
  ```c
  void mantiq_panic_at(const char* message, const char* file, int line, int col);
  ```
- **Static Strings**: The compiler defines a global constant for the source file name (e.g. `@.str_file = private unnamed_addr constant [8 x i8] c"main.mq\00"`).
- **Execution**: When a check fails, the compiler passes the file constant along with the line and column numbers of the failing expression.
- **Trace Output**: The program prints a detailed, user-friendly diagnostic before termination:
  `Runtime Panic: Index out of bounds at main.mq:14:10`

## Implementation Implications
- The LLVM IR has clear boundaries and visual anchors for every high-level statement.
- Runtime panics do not require GDB/LLDB to trace back to their source origin, saving debugging time.
